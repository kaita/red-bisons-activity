# RED BISONS 活動管理ブラウザアプリ 仕様ドラフト

## 目的

RED BISONS の活動予定、選手の参加可否、保護者の鍵開け・見守り・鍵閉め、引き継ぎ事項を、保護者がスマホ/PCブラウザから確認・入力できるようにする。

現行の Google スプレッドシート「Red Bisons 2026-27 活動シート」を元データとして、入力しやすさ、未回答の把握、当番不足の可視化を改善する。

## 元データ

- スプレッドシート: Red Bisons 2026-27 活動シート
- タブ:
  - 活動スケジュール
  - 情報
- 活動スケジュールの構造:
  - 1活動が3行で構成される
  - 1行目: 日付、参加者数、保護者当番集計、選手参加者一覧、各選手の参加可否
  - 2行目: 時間、見守り/鍵閉め集計、保護者当番チェック
  - 3行目: 場所、引き継ぎ、各家庭の遅刻早退・引率コメント
- 参加可否の選択肢:
  - 参加
  - 欠席
  - 未回答
  - (未定)
- 保護者当番:
  - 鍵開け・見守り
  - 見守り・鍵閉め
- 情報タブ:
  - RED BISONS サイト URL
  - 2026-27 ルールブック URL

## 想定ユーザー

- 保護者
  - 自分の子どもの参加可否を入力する
  - 鍵開け、鍵閉め、見守りに協力できる日と時間を入力する
  - 遅刻、早退、引率、その他コメントを記入する
  - 活動予定と当日の引き継ぎを確認する
  - 他の保護者と活動単位で必要な連絡を取る
- 管理者
  - 活動予定を登録・編集する
  - メンバーを管理する
  - 未回答者、当番不足、参加人数を確認する
  - 保護者向けの案内に必要な情報を一覧化する

## MVPスコープ

### 認証・セキュリティ

- 保護者・管理者ともにGoogleログインを必須にする
- 未ログイン状態では活動情報を表示しない
- 管理者権限はサーバー側で判定する
- 保護者は自分に紐づく子どもと自分の見守り情報だけを編集できる
- 公開GitHubリポジトリに秘密情報を置かない
- Google Sheets、Google Calendar、Google OAuthの認証情報は環境変数またはホスティング環境のSecretで管理する

### 保護者向け

- 活動一覧を日付順に表示する
- 各活動に以下を表示する
  - 日付
  - 時間
  - 場所
  - 参加人数
  - 参加予定選手
  - 鍵開け担当
  - 鍵閉め担当
  - 見守り可能な保護者と時間帯
  - 引き継ぎ事項
- 自分の子どもについて以下を入力できる
  - 参加可否
  - 鍵開け可否
  - 鍵閉め可否
  - 見守り可否
  - 見守り可能な開始時刻
  - 見守り可能な終了時刻
  - 遅刻早退・引率コメント
- 未回答の活動を目立たせる
- 当番が0人の活動を目立たせる
- 見守りの大人が少ない時間帯を目立たせる
- 見守りの大人がいない活動・時間帯には注意マークを表示する
- スマホから少ないタップで回答できる
- Googleカレンダーに活動予定を登録しやすくする

### 管理者向け

- 活動の新規追加・編集
  - 日付
  - 時間
  - 場所
  - 引き継ぎ
- 活動日時の登録・変更は管理者のみ可能
- メンバー一覧の管理
  - 選手名
  - 学年
  - 保護者/家庭単位の表示名
- 活動ごとの集計確認
  - 参加者数
  - 参加者一覧
  - 未回答者一覧
  - 当番担当者一覧
  - 当番不足の活動一覧
  - 見守り人数が不足する時間帯

## 機能仕様

### メンバー仕様

- 登録できるユーザー種別:
  - 管理者
  - 保護者
- 管理者は以下を登録・編集できる:
  - 活動
  - 選手
  - 保護者
  - 選手と保護者の紐づけ
- 保護者は以下を編集できる:
  - 自分に紐づく子どもの参加可否
  - 自分の鍵開け可否
  - 自分の鍵閉め可否
  - 自分の見守り可否と時間帯
  - 自分のコメント
- 保護者は他家庭の回答状況を閲覧できるが、編集はできない
- 1人の保護者が複数の子どもに紐づく可能性を許容する
- 1人の子どもに複数の保護者が紐づく可能性を許容する

### 活動仕様

- 管理者は活動を登録・編集できる
- 活動に持たせる情報:
  - 日付
  - 開始時刻
  - 終了時刻
  - 場所
  - 引き継ぎ事項
  - 公開状態
- 保護者は活動ごとに子どもの出欠を入力できる
- 出欠ステータス:
  - 参加
  - 欠席
  - 未回答
  - 未定
- 活動日時が変更された場合、既存回答の扱いを明確にする
  - 初期案: 回答は維持し、活動詳細に「日時変更あり」を表示する

### 見守り仕様

- 鍵開け、鍵閉め、見守り可否は別々にチェックできる
- 鍵開け:
  - 活動開始前に鍵を開けられる場合にチェックする
- 鍵閉め:
  - 活動終了後に鍵を閉められる場合にチェックする
- 見守り:
  - 見守り可能な場合にチェックする
  - 見守り開始時刻と終了時刻を入力する
  - 時刻は活動時間の範囲内を基本とする
  - 部分参加を許容する
- 活動詳細では、見守り可能な大人を時間帯ごとに可視化する
- 見守り人数が0人または不足する時間帯を強調表示する
- 保護者向け表示では、見守りが足りない状態を注意マークで分かりやすく示す
- 注意マークの初期条件:
  - 鍵開け担当が0人
  - 鍵閉め担当が0人
  - 見守り人数が0人の時間帯がある
  - 見守り必要人数を下回る時間帯がある

### 表示仕様

- カレンダー的表示を提供する
  - 月表示
  - 週表示またはリスト表示
  - 直近活動の一覧
- 活動詳細を表示する
  - 活動日時
  - 場所
  - 参加予定者
  - 欠席者
  - 未回答者
  - 鍵開け担当
  - 鍵閉め担当
  - 見守り可能な保護者と時間帯
  - 引き継ぎ事項
  - コメント
- 見守り可視化:
  - 活動時間を30分単位などの枠で分割する
  - 各時間帯に見守り可能な大人の人数を表示する
  - 人数が足りない時間帯を色で目立たせる
- 参加者一覧を表示する
  - 参加予定の子ども
  - 欠席予定の子ども
  - 未回答の子ども
  - 必要に応じて保護者名または家庭名

### コミュニケーション仕様

- 活動ごとに保護者同士の連絡をしやすくする
- MVPでは以下のいずれかを候補とする:
  - 活動ごとのコメント欄
  - 引き継ぎ欄への追記
  - 外部連絡手段へのリンク
- コメントは活動詳細に表示する
- 通知機能はMVPでは候補扱いとし、必要性を確認してから実装する

### Googleカレンダー連携仕様

- 活動予定をGoogleカレンダーに反映できるようにする
- MVPでは低コスト・低運用負荷を優先し、RED BISONS共有Googleカレンダーを購読してもらう方式を採用する
- アプリ側では以下を提供する:
  - 共有カレンダーの購読導線
  - 各活動の「Googleカレンダーに追加」リンク
  - 活動詳細から共有カレンダー予定を確認できる導線
- 将来拡張として、保護者または子どものGoogleアカウントに予定を書き込む方式を検討する
- カレンダー予定に含める情報:
  - 活動日時
  - 場所
  - 参加者一覧へのリンク
  - 活動詳細へのリンク
  - 引き継ぎ概要
- 子ども本人のカレンダーにも予定を入れる場合:
  - MVPでは共有カレンダーを子どものカレンダーから購読する運用を基本とする
  - 将来拡張では、子どものGoogleアカウントまたは共有先メールアドレスを登録できる
  - 将来拡張では、参加予定の活動だけを予定に入れるか、全活動を入れるか選べるようにする
  - 個人カレンダーへの直接書き込みは認証・権限・同意が必要なため、MVPでは扱わない

## 画面案

### 1. 活動一覧

- カレンダー表示とリスト表示を切り替えられる
- 直近の活動をすぐ確認できる
- フィルタ:
  - 未回答のみ
  - 参加予定のみ
  - 当番不足のみ
  - 見守り不足のみ
- 活動カードまたはテーブルで概要を表示
- 見守り不足がある活動には注意マークを表示する
- Googleカレンダーに追加できる導線を表示する

### 2. 活動詳細

- 活動情報
- 参加者一覧
- 鍵開け担当
- 鍵閉め担当
- 見守り時間帯の可視化
- 引き継ぎ事項
- 自分の回答フォーム
- 活動単位のコメント/連絡欄
- Googleカレンダー登録リンク
- 見守り不足がある時間帯の注意表示

### 3. 自分の回答

- 活動ごとに参加可否と当番可否をまとめて編集
- スマホで短時間に入力できるUIを優先する
- 参加可否は大きめの選択ボタンで入力する
- 鍵開け、鍵閉め、見守りは独立したチェックボックスまたはトグルで入力する
- 見守り時間はスマホで操作しやすい時刻選択UIにする
- 活動時間外の時刻を選びにくいUIにする

### 4. 管理画面

- 活動編集
- メンバー編集
- 未回答・当番不足の一覧
- 見守り不足時間帯の一覧
- Googleカレンダー同期状態の確認

## UX方針

- スマホファーストで設計する
- 保護者の入力画面では、1活動あたりの入力を短時間で完了できるようにする
- ボタン、トグル、時刻入力は指で押しやすいサイズにする
- 活動一覧では、未回答、当番不足、見守り不足を視覚的に判別しやすくする
- 管理者向け画面は、一覧性と絞り込みを優先する
- 親同士の連絡に必要な情報を活動詳細に集約する
- 見守り不足は、単なる数字だけでなく注意マークと短い文言で行動につながる表示にする
- カレンダー登録は、保護者が迷わないように活動詳細から1タップで到達できるようにする

## 技術方針の候補

### 案A: Google Sheetsを当面のデータベースにする

- 現行シートと同期しやすい
- 初期導入が軽い
- 既存運用を残せる
- 同時編集や権限設計、履歴管理には注意が必要

### 案B: アプリ専用DBに移行する

- ログイン、権限、履歴、通知を作りやすい
- 将来的な拡張に強い
- 既存シートから初期データ移行が必要
- シート運用との二重管理を避ける設計が必要

### 低コスト運用方針

- 更新頻度は高くない前提で、常時稼働サーバーをできるだけ避ける
- MVPでは静的フロントエンド + サーバーレスAPI + Google Sheets/軽量DBを候補にする
- Googleカレンダー連携は、共有カレンダー購読と追加リンク中心にして、個人カレンダーへの直接書き込みは後回しにする
- 通知や自動同期は、必要なタイミングだけ実行する方式を優先する
- まずは無料枠または低額枠で運用できる構成を前提にする
- 将来的に利用者数や更新頻度が増えた場合だけ、専用DBやバックグラウンド同期を強化する

### 公開リポジトリ前提のセキュリティ方針

- リポジトリはGitHubの `kaita` アカウント配下で公開リポジトリとして作成する
- OAuthクライアントID、サービスアカウントメール、秘密鍵、カレンダーID、スプレッドシートIDは環境変数で管理する
- クライアント側にはGoogleログイン用の公開可能なClient ID以外の秘密情報を渡さない
- Google IDトークンはAPI側で署名、audience、有効期限、issuerを検証する
- Google SheetsとGoogle Calendarへの書き込みはAPI側だけが行う
- CORSは原則同一オリジンに限定する
- エラー応答に内部設定値やスタックトレースを含めない

## 受入基準案

- 保護者と管理者がGoogleログインなしではアプリを利用できない
- 保護者がスマホブラウザから自分の子どもの参加可否を更新できる
- 保護者が鍵開け、鍵閉め、見守り可否を別々に入力できる
- 保護者が見守り可能な開始時刻と終了時刻を入力できる
- 管理者が活動ごとの参加人数、未回答者、当番不足を確認できる
- 管理者だけが活動日時を登録・編集できる
- 現行シートの主要情報をアプリ画面で表現できる
- 当番が0人の活動が一目で分かる
- 見守り人数が不足する時間帯が一目で分かる
- 見守りの大人が足りない活動に注意マークが表示される
- 活動予定をGoogleカレンダーへ登録または同期できる
- 活動詳細で参加者一覧を確認できる
- 引き継ぎ事項と家庭別コメントを活動ごとに確認できる
- スマホで活動確認と回答更新が無理なく操作できる
- 低コスト運用を前提に、常時稼働サーバーなしでも成立する構成を選べる

- 通知の要否
- 公開先とホスティング方法
- 既存シートへの書き戻し要否
- 見守りの必要人数
- 見守り可視化の時間単位
- 活動コメントをアプリ内に持つか、外部連絡手段へ寄せるか
- 保護者招待・初回登録の流れ
- 活動日時変更時に回答を維持するか、再回答扱いにするか
- 子どものカレンダーには全活動を入れるか、参加予定の活動だけを入れるか
- カレンダー予定に参加者一覧を直接載せるか、アプリ詳細へのリンクに留めるか
- 見守り不足の注意マークを出す基準
